Weekly Nerd

Killian Valkhof Weekly Nerd 1

Rule of Least Power

Gebruik niet automatisch JavaScript voor alles. Veel functionaliteiten kunnen tegenwoordig gewoon met HTML en CSS worden opgelost.

Volgens de Rule of Least Power gebruik je eerst HTML, daarna CSS en pas daarna JavaScript.

  • HTML voor structuur.
  • CSS voor presentatie en interactie.
  • JavaScript alleen wanneer het echt nodig is.

JavaScript is krachtig, maar ook zwaarder en kwetsbaarder dan HTML en CSS.

Polypane

Polypane is een tool van Kilian Valkhof die helpt bij responsive design.

  • Gratis voor studenten.
  • Gebouwd met JavaScript.

Dat maakt duidelijk dat hij niet tegen JavaScript is, maar wel tegen onnodig gebruik ervan.

React en JSX

Voor React-developers is het goed om te onthouden dat JSX eigenlijk een dialect van HTML is.

Custom switches

Veel developers bouwen switches met div-elementen en JavaScript-events, terwijl HTML en CSS hiervoor vaak voldoende zijn.

  • Gebruik een checkbox als basis.
  • Gebruik appearance: none om de standaardstijl te verwijderen.
  • Gebruik ::before voor de eigen vormgeving.
  • Gebruik :checked voor de actieve staat.
  • Gebruik :focus voor toetsenbordnavigatie.

Verwijder nooit zomaar een outline zonder een toegankelijk alternatief aan te bieden.

Safari heeft zelfs een ingebouwde switch-weergave.

Smooth scrolling en page transitions

Functionaliteiten die vroeger vaak met jQuery werden gebouwd, kunnen nu grotendeels met CSS worden opgelost.

  • scroll-behavior: smooth voor vloeiend scrollen.
  • prefers-reduced-motion voor gebruikers die gevoelig zijn voor animaties.

Accordions

Voor uitklapbare content kunnen de elementen <details> en <summary> worden gebruikt.

  • <details> bevat de verborgen content.
  • <summary> is de altijd zichtbare titel.
  • Werkt zonder JavaScript.
  • Is standaard toegankelijker.
  • Kan volledig worden gestyled met CSS.

Het standaard pijltje is aanpasbaar via ::marker.

Dialog

Het <dialog>-element kan worden gebruikt voor modals.

  • Standaard verborgen.
  • Vervangt veel JavaScript-oplossingen.
  • Kan worden geanimeerd met @starting-style.

Soms is nog een klein beetje JavaScript nodig voor het openen en sluiten.

Field sizing

Met field sizing kunnen invoervelden automatisch meegroeien met hun inhoud.

Deze functionaliteit wordt momenteel ondersteund door Chrome en Safari.

Select styling

Select-elementen waren lange tijd moeilijk te stylen zonder functionaliteit kwijt te raken.

Nieuwe CSS-mogelijkheden maken dit eenvoudiger.

  • ::picker(select)
  • appearance: base-select

Op dit moment is de ondersteuning vooral aanwezig in Chrome.

Scroll-driven animations

Scroll-animaties kunnen tegenwoordig direct aan scrollgedrag worden gekoppeld met CSS.

Hierdoor is JavaScript voor veel scroll-animaties niet meer nodig.

Belangrijkste takeaway

Gebruik JavaScript alleen wanneer het daadwerkelijk nodig is.

Moderne HTML en CSS bieden veel meer mogelijkheden dan veel developers zich realiseren.

Peter Paul Koch Weekly Nerd 2

Wat is een browser?

Een browser bestaat uit:

  • HTML
  • CSS
  • DOM (Document Object Model) - Kopie van HTML geschikt voor JavaScript.
  • AOM (Accessibility Object Model) - DOM met functies voor screenreaders.
  • Interface
  • JavaScript engine

Parser

Een parser is een softwarecomponent dat invoer ontleedt en omzet in een gestructureerd format.

Rendering engine

De rendering engine is verantwoordelijk voor parsing en rendering van HTML en CSS, en voor de constructie van DOM- en AOM-trees.

De rendering engine is niet verantwoordelijk voor de interface of de JavaScript engine.

JavaScript en DOM

  • Core is wat JavaScript een programmeertaal maakt.
  • DOM is de manier waarop JavaScript HTML kan aanspreken en wijzigen.

Render Blocking

CSS en JavaScript blokkeren rendering, tenzij JavaScript async of defer gebruikt.

  • Do it now: script met src
  • Do it later: script met defer
  • Do it whenever: script met async

Backwards Compatibility

Principe: als iets in een browser heeft gewerkt, moet het altijd in die browser blijven werken.

zorgt ervoor dat de browser de moderne rendering engine gebruikt.

Design tools door de jaren heen

Denk in onderdelen, niet pixels

Kijk niet alleen naar individuele pixels, maar naar grotere onderdelen van een ontwerp.

  • Focus op structuur.
  • Focus op herbruikbare componenten.
  • Niet op perfecte pixel placement.

CSS Grid en whitespace

De fr-unit binnen CSS Grid kan worden gebruikt om beschikbare ruimte beter te verdelen.

Hierdoor ontstaat meer flexibiliteit binnen layouts.

Ook witruimte kan hiermee beter worden gecontroleerd.

Subgrid

Child-elementen kunnen hierdoor dezelfde gridstructuur volgen als hun parent.

Dit helpt bij het bouwen van consistente layouts.

Handige selectors

Denk meer in systemen, layouts en componenten dan in losse pixels.

Robbert en Yolijn Weekly Nerd 4

Formulieren

Een formulier moet een duidelijke en logische flow hebben.

  • Introductie en uitleg.
  • Benodigde gegevens invullen.
  • Gegevens controleren.
  • Foutmeldingen tonen na verzenden.
  • Duidelijke bevestiging na succesvolle verzending.

Introductie

Maak vooraf duidelijk wat gebruikers kunnen verwachten.

  • Hoe lang het formulier ongeveer duurt.
  • Hoeveel stappen er zijn.
  • Of aanvullende documenten nodig zijn.

Een tijdsinschatting moet altijd realistisch zijn.

Validatie en feedback

Feedback moet duidelijk zijn en gebruikers helpen om fouten op te lossen.

Bij fouten:

  • Rode outlines rondom velden.
  • Duidelijke foutmeldingsteksten.
  • Een samenvatting van alle fouten na verzenden.

Bij succes:

  • Groene feedback.
  • Een duidelijke bevestiging.
  • Uitleg over vervolgstappen.

Vraag alleen noodzakelijke informatie

Vraag niet meer informatie dan nodig is.

  • Geslacht is vaak niet noodzakelijk.
  • Vraag geen e-mailadres meerdere keren.
  • Vermijd onduidelijke afkortingen.
  • Leg uit waarom informatie nodig is.

Labels en inputvelden

Labels spelen een belangrijke rol in gebruiksvriendelijkheid en toegankelijkheid.

  • Plaats labels boven invoervelden.
  • Koppel labels aan inputs zodat ze klikbaar zijn.
  • Houd labels kort en duidelijk.
  • Gebruik placeholders niet als vervanging van labels.
  • Gebruik liever "verplicht" of "niet verplicht" dan alleen een sterretje.

Voornaam is duidelijker dan Voorna(a)m(en).

Design richtlijnen

  • Duidelijke kleurcontrasten.
  • Klikbare elementen minimaal 44 × 44 pixels.
  • Buttons aan het begin van een regel plaatsen.
  • Voortgang altijd bovenaan tonen.
  • Consistente navigatie gebruiken.

Bijvoorbeeld: volgende, vorige en annuleren.

Gebruiksgemak

Maak gebruik van bestaande browserfunctionaliteiten waar mogelijk.

  • Automatisch invullen ondersteunen.
  • Copy en paste toestaan.
  • Browser autocomplete ondersteunen.
  • Voldoende ruimte laten voor autocomplete-menu's.
  • Belangrijke uitleg zichtbaar houden.

aria-describedby kan worden gebruikt voor aanvullende uitleg.

Controle stap

Laat gebruikers hun gegevens controleren voordat ze definitief verzenden.

Hierdoor kunnen fouten worden ontdekt voordat de aanvraag wordt verstuurd.

CSS sizing

Gebruik liever logische CSS-properties dan fysieke afmetingen.

  • inline-size in plaats van width.
  • block-size in plaats van height.

Flexibele validatie

Validatie moet gebruikers helpen en niet onnodig beperken.

  • Geef maximale aantallen tekens duidelijk aan.
  • Voorkom onnodige restricties.
  • Accepteer meerdere invoerformaten waar mogelijk.

Bijvoorbeeld een postcode met of zonder spatie.

Toetsenbord toegankelijkheid

Een formulier moet volledig zonder muis te gebruiken zijn.

  • Tab.
  • Shift + Tab.
  • Pijltjestoetsen.

Daarnaast zijn een logische tabvolgorde en duidelijke focus states belangrijk.

Radio buttons moeten ook met de pijltjestoetsen bedienbaar zijn.

Screenreaders en toegankelijkheid

Toegankelijkheid moet worden getest met echte gebruikers.

Screenreaders moeten kunnen voorlezen:

  • Labels.
  • Beschrijvingen.
  • Verplichte velden.
  • Foutmeldingen.
  • Formulierstappen.
  • Bevestigingen na verzending.

Feedback moet behulpzaam zijn en niet storend werken.

Belangrijkste takeaway

Goede formulieren vragen alleen noodzakelijke informatie en begeleiden gebruikers stap voor stap door het proces.

Toegankelijkheid, duidelijke feedback en gebruiksgemak zijn daarbij minstens zo belangrijk als het visuele ontwerp.

Rosa

Centraal stond de vraag hoe afhankelijk we zijn geworden van grote technologiebedrijven en hoe we daar bewuster mee om kunnen gaan.

Planned obsolescence

Veel apparaten worden steeds moeilijker te repareren of voelen sneller verouderd aan.

Gebruikers hebben vaak weinig inzicht in of controle over deze systemen.

Een server is in de basis:

  • Een computer.
  • Verbonden met een router.
  • Slaat informatie op.
  • Verstuurt informatie naar gebruikers.

Duurzamere hosting

Digitale onafhankelijkheid begint bij begrijpen hoe technologie werkt en waar onze afhankelijkheden liggen.

Johan Huijkman Weekly Nerd 6

Toegankelijkheid

"Je gaat het pas zien als je het doorhebt."

"Je hoort het pas als je luistert."

De kern van de talk was dat toegankelijkheid pas echt begrijpelijk wordt wanneer je luistert naar mensen die er dagelijks mee te maken hebben.

Richtlijnen zijn waardevol, maar ervaringen van echte gebruikers zijn vaak nog waardevoller.

WCAG-principes

WCAG is gebaseerd op vier kernprincipes.

Perceivable
Content moet waarneembaar zijn.
Operable
Een website moet bedienbaar zijn.
Understandable
Informatie moet begrijpelijk zijn.
Robust
Moet werken op verschillende technologieën en hulpmiddelen.

WCAG is een hulpmiddel, geen einddoel.

Toegankelijkheid moet ook worden getest met echte gebruikers.

Screenreaders en alt-tekst

Maak geen aanpassingen op basis van aannames.

Praat met mensen die dagelijks screenreaders gebruiken en leer hoe zij websites ervaren.

Veel problemen zijn al opgelost binnen bestaande hulpmiddelen.

Niet iedere afbeelding heeft uitgebreide alt-tekst nodig.

  • Informatieve afbeeldingen krijgen een beschrijving.
  • Decoratieve afbeeldingen mogen een lege alt-tekst hebben.

Bijvoorbeeld: alt="".

Martijn — kan bijna niets zien

Martijn vertelde over problemen die hij regelmatig tegenkomt tijdens het gebruik van websites.

Veelvoorkomende problemen:

  • Knoppen zonder duidelijke hover- of focus states.
  • Cookie banners waarbij scrollen niet werkt.
  • Slecht kleurcontrast.
  • Captcha's.

Wat helpt:

  • Korte en duidelijke teksten.
  • Schaalbare content.
  • Goed contrast.
  • Logische informatiegroepering.
  • Duidelijke focus indicatoren.

Nauduah — beperkte mobiliteit

Nauduah vertelde over uitdagingen die ontstaan door beperkte mobiliteit en overbelasting door interfaces.

Veelvoorkomende problemen:

  • Felle kleuren.
  • Te veel informatie tegelijk.
  • Drukke interfaces.
  • Lang moeten zoeken naar informatie.

Wat helpt:

  • Duidelijke structuur.
  • Informatie groeperen.
  • Visueel rustige websites.
  • Dark mode.
  • Gegevens kunnen opslaan.

Guido — autisme

Guido vertelde hoe websites soms onnodig veel prikkels veroorzaken.

Veelvoorkomende problemen:

  • Te veel visuele prikkels.
  • Te veel audio-prikkels.
  • Harde advertenties.
  • Te veel informatie tegelijk.
  • Onduidelijke formuleringen.

Wat helpt:

  • Rustige layouts.
  • Voorspelbaarheid.
  • Duidelijke instructies.
  • Concrete teksten.
  • Weinig afleiding.

Toepassing op eigen project

Denk na over waarom gebruikers mogelijk afhaken tijdens het gebruik van je eigen project.

Verschillende gebruikers ervaren verschillende barrières.

Door gebruikers te betrekken en naar hun ervaringen te luisteren kunnen toegankelijkheidsproblemen eerder worden ontdekt en opgelost.

Belangrijkste takeaway

Toegankelijkheid gaat niet alleen over richtlijnen, maar vooral over mensen.

Je begrijpt toegankelijkheid pas echt wanneer je luistert naar gebruikers die dagelijks afhankelijk zijn van hulpmiddelen en aangepaste workflows.

Reflectie

Reflectie begin

Begin van de minor

Aan het begin van de minor lag mijn focus vooral op het leren van de technische basis.

Ik had ideeën die vaak groter waren dan wat op dat moment realistisch was, zoals het bouwen van een website met een geanimeerd 3D-model.

Al snel leerde ik dat het belangrijker is om eerst een werkende basis neer te zetten voordat je extra functionaliteiten toevoegt.

Tijdens Sprint 0 en de eerste projecten heb ik veel tijd besteed aan HTML, CSS, responsive design en het begrijpen van hoe verschillende onderdelen van een website samenwerken.

Ik was vaak bezig met uitzoeken hoe iets technisch werkte en gebruikte opdrachten vooral als manier om nieuwe technieken te leren.

Daarbij merkte ik dat ik soms snel geneigd was om oplossingen op te zoeken, terwijl ik juist meer leer wanneer ik eerst zelf probeer een probleem op te lossen.

In deze periode draaide mijn ontwikkeling vooral om het opbouwen van technische vaardigheden en het krijgen van vertrouwen in frontend development.

Reflectie midden

Midden van de minor

Halverwege de minor begon mijn focus steeds meer te verschuiven van techniek naar de gebruiker.

Tijdens vakken zoals HCD en Browser Technologies kwam ik vaker in contact met onderwerpen als toegankelijkheid, progressive enhancement en gebruikersonderzoek.

Door samen te werken met gebruikers zoals Ihab ontdekte ik dat een technisch werkende oplossing niet automatisch een goede oplossing is.

Feedbackmomenten werden steeds belangrijker en ik merkte dat mijn projecten beter werden wanneer ik gebruikers actief betrok bij het proces.

Ook begon ik bewuster na te denken over keuzes rondom JavaScript, HTML en CSS.

In plaats van direct naar een technische oplossing te grijpen, keek ik vaker naar wat de meest geschikte oplossing was.

Daarnaast kreeg ik meer inzicht in hoe browsers werken, hoe formulieren toegankelijk gemaakt kunnen worden en waarom standaarden vaak bestaan met een goede reden.

Hierdoor ontwikkelde ik niet alleen mijn technische vaardigheden verder, maar ook mijn vermogen om vanuit de gebruiker te denken.

Daarnaast heb ik gewerkt aan de backend met Supabase. Hiervoor heb ik onder andere authenticatie, gebruikersvoortgang en het opslaan van cursusdata ontwikkeld.

Gedurende het project merkte ik hoe belangrijk gebruikerstesten zijn. Verschillende onderdelen die voor ons logisch leken, bleken voor gebruikers minder duidelijk. Door feedback te verzamelen en te verwerken werd de cursus overzichtelijker en toegankelijker.

Dit project heeft mij niet alleen meer ervaring gegeven met Astro, databases en authenticatie, maar ook laten zien dat techniek uiteindelijk in dienst staat van de gebruiker.

Belangrijkste takeaway

Een technisch goede oplossing is pas echt succesvol wanneer gebruikers zonder uitleg begrijpen hoe het werkt en hun doel kunnen bereiken.

Reflectie eind

Einde van de minor

Aan het einde van de minor kijk ik anders naar webdevelopment dan aan het begin.

Waar ik eerst vooral bezig was met het bouwen van functionaliteiten, denk ik nu veel meer na over de gevolgen van ontwerp- en technische keuzes.

Onderwerpen zoals toegankelijkheid, duurzaamheid, digitale onafhankelijkheid, performance en gebruikerservaring spelen nu een grotere rol in mijn afwegingen.

De Weekly Nerds, de Smashing Conference en de verschillende projecten hebben mij laten zien dat webdevelopment veel meer is dan alleen code schrijven.

Ik ben kritischer geworden op het gebruik van frameworks, JavaScript en externe diensten, en kijk vaker naar wat een browser van zichzelf al kan.

Daarnaast heb ik geleerd om eerder te testen, feedback serieuzer te nemen en iteratief te werken.

Ik sluit de minor af met aanzienlijk meer technische kennis dan waarmee ik begon, maar vooral met een bredere blik op het vakgebied en een beter begrip van de mensen voor wie we uiteindelijk websites bouwen.
